# Penetration Test Report: Akademi App (Live Environment)

**Target:** `https://akademi-app.onrender.com`
**Date:** April 14, 2026
**Tester:** Jules (AI Security Auditor)

---

## Executive Summary
A Dynamic Application Security Testing (DAST) was performed on the live Akademi backend. Several critical vulnerabilities were discovered, most notably **Broken Access Control (IDOR)** and **Persistent Cross-Site Scripting (XSS)**. These issues could allow an attacker to view private student data and execute malicious scripts in other users' browsers.

---

## 1. Critical Vulnerabilities

### 1.1 Broken Access Control (IDOR) on Sessions
*   **Endpoint:** `GET /sessions/:id`
*   **Severity:** **Critical**
*   **Description:** The application fails to verify if the authenticated user owns the session they are requesting. An attacker with a valid student token can view the details and message history of any session if they know or guess the UUID.
*   **Proof of Concept:**
    1.  Log in as User A and create a session (ID: `0e912525-...`).
    2.  Log in as User B.
    3.  Request `GET /sessions/0e912525-...` using User B's token.
    4.  **Result:** User B successfully retrieves User A's private session data.
*   **Impact:** Unauthorized access to private academic conversations and session history.

### 1.2 Persistent Cross-Site Scripting (XSS) in User Profiles
*   **Endpoint:** `PATCH /users/me`
*   **Severity:** **High**
*   **Description:** The `name` field (and potentially others) does not sanitize input. An attacker can inject a malicious script that will execute when an admin views their profile in the management dashboard or when other users interact with their shared content.
*   **Proof of Concept:**
    *   Payload: `{"name": "Pentester <script>alert('XSS')</script>"}`
    *   The backend stores this string literally and returns it in subsequent API calls.
*   **Impact:** Session hijacking, redirection to malicious sites, or unauthorized actions on behalf of other users (especially admins).

## 2. Medium Vulnerabilities

### 2.1 Verbose Error Messages (Information Disclosure)
*   **Severity:** **Medium**
*   **Description:** When a database operation fails (e.g., due to invalid data types), the backend returns the full Prisma error message, including internal file paths and table schema details.
*   **Example:** `Invalid db_1.default.user.create() invocation... Argument level: Invalid value provided. Expected Int, provided String.`
*   **Impact:** Leaks internal implementation details, making it easier for an attacker to craft precise exploits.

### 2.2 User Enumeration via Registration
*   **Endpoint:** `POST /auth/register`
*   **Severity:** **Medium**
*   **Description:** The API returns distinct messages for "Email already registered" versus success.
*   **Impact:** Allows an attacker to confirm if a specific person (e.g., a known email address) has an account on the platform.

---

## 3. Low Vulnerabilities & Best Practices

### 3.1 Brute-Force Susceptibility on OTP
*   **Endpoint:** `/auth/verify-email`
*   **Severity:** **Low/Medium**
*   **Description:** While `/auth/login` has a rate limiter, `/auth/verify-email` and `/auth/resend-verification` do not appear to have specific limits in the route configuration.
*   **Impact:** Allows an attacker to potentially brute-force the 6-digit verification token within its 15-minute validity window.

---

## Recommendations

1.  **Enforce Ownership Checks:** Update all service methods (e.g., `sessionsService.getSession`) to include the `userId` in the query: `prisma.session.findUnique({ where: { id, user_id: userId } })`.
2.  **Sanitize Input:** Use a library like `dompurify` or `xss` on the backend to clean all user-provided strings before saving them to the database.
3.  **Generic Error Handling:** Implement a global error handler that catches database errors and returns a generic "Internal Server Error" message to the client, while logging the details to Sentry.
4.  **Secure Authentication Flow:** Ensure all authentication-related endpoints (OTP verification, password reset, etc.) are protected by the same strict rate-limiting as the login endpoint.
